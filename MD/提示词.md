本次重点修以下问题：

一、修复 Vite 本地代理

问题：
前端 axios baseURL 是 /api，后端也使用 /api context-path。
本地开发时必须把 /api 代理到 http://localhost:8080，不要 rewrite。

要求：
1. 修改 vite.config.ts。
2. 增加 server.proxy。
3. /api 转发到 http://localhost:8080。
4. 不要 rewrite /api，因为后端已有 /api context-path。
5. 保留现有 alias 和插件配置。

建议配置：
server: {
  proxy: {
    '/api': {
      target: 'http://localhost:8080',
      changeOrigin: true
    }
  }
}

二、检查并修复 request.ts

要求：
1. 确认 axios baseURL 仍然是 /api。
2. 确认请求头使用 Authorization: Bearer <token>。
3. 确认后端返回 code === 0 时前端正常取 data。
4. 401 时清除登录态并跳转登录页。
5. 403 时给出“无权限访问”提示，不要直接白屏。
6. 不要破坏下载、导出、预览等 blob 或特殊响应处理。

三、修复后台知识库管理页筛选参数

后端现在支持：
GET /admin/knowledge-bases
参数：
- keyword
- username
- status
- pageNo
- pageSize

要求：
1. 检查前端后台知识库 API 方法。
2. 确保请求参数包含 keyword、username、status、pageNo、pageSize。
3. 检查后台知识库管理页面的筛选表单。
4. 如果页面已有“创建人/用户名”和“状态”筛选，必须接入真实请求参数。
5. 如果页面没有对应筛选项，请补齐最小筛选 UI。
6. 不要新增审核、推荐、分享等扩展功能。

四、修复用户侧文档管理页

后端现在支持：
GET /documents
参数：
- knowledgeBaseId，可选
- keyword
- parseStatus
- embeddingStatus
- pageNo
- pageSize

要求：
1. 检查前端 getDocumentPage API 方法。
2. 确保 knowledgeBaseId 是可选参数。
3. 确保 parseStatus、embeddingStatus 能正确传给后端。
4. 文档管理页不能因为未选择知识库就不查询。
5. 如果选择了知识库，则带 knowledgeBaseId 查询。
6. 如果未选择知识库，则查询当前用户全部文档。
7. 解析状态枚举值必须使用后端值：
   - PENDING
   - PARSING
   - SUCCESS
   - FAILED
8. 向量状态枚举值必须使用后端值：
   - PENDING
   - PROCESSING
   - SUCCESS
   - FAILED
9. 不要使用中文、小写、done、ok 等非后端枚举值。

五、修复后台文档管理页

后端现在支持：
GET /admin/documents
参数：
- keyword
- knowledgeBaseId
- parseStatus
- fileType
- pageNo
- pageSize

要求：
1. 检查 getAdminDocuments API 方法。
2. 确保参数名与后端一致。
3. 后台文档管理页面筛选项至少包括：

六、修复后台任务管理页

后端现在支持：
GET /admin/document-tasks
参数：
- status
- keyword
- pageNo
- pageSize

后端允许前端传 taskType，但当前会忽略。

要求：
1. 检查 getDocumentTasks API 方法。
2. 确保 status、keyword、pageNo、pageSize 正确传参。
3. taskType 如果页面已有，可以暂时保留，但要避免用户误以为筛选生效。
   可选做法：
   - 隐藏 taskType 筛选；
   - 或保留但标注“暂未启用”；
   - 或从请求参数里去掉 taskType。
4. 检查任务列表字段展示：
   - id
   - documentId
   - documentName
   - status
   - failReason
   - createdAt
   - startedAt
   - finishedAt
5. 字段不存在时要做空值兜底，不能导致页面报错。

七、接入后台任务重试按钮

后端现在支持：
POST /admin/document-tasks/{id}/retry

要求：
1. 检查 retryDocumentTask API 方法。
2. 确认路径是 /admin/document-tasks/{id}/retry。
3. 在后台任务列表或详情页接入重试按钮。
4. 只有 FAILED 状态任务显示或启用重试按钮。
5. 点击重试前必须二次确认。
6. 请求成功后刷新任务列表或任务详情。
7. 请求失败时显示后端错误信息。
8. 不要自己在前端模拟重试状态。

# 前后端接口联调差异清单（重制版）

> 项目：KnowFlow AI 个人知识库与智能文档问答平台  
> 分析对象：
> - 后端仓库：`https://github.com/wbn-2020/ai-knowledge-java.git`
> - 前端仓库：`https://github.com/wbn-2020/ai-knowledge-vue.git`
> - PRD：`知识库PRD_V1.md`
>
> 说明：本文件只做静态分析，不修改代码。重点对照前端 `src/api`、`src/router`、`src/views` 与后端 Controller，并按 PRD 的 MVP 范围判断是否满足联调闭环。

---

## 0. 总体结论

当前项目不是“完全没联通”，而是：

1. **主链路接口基本存在**：登录注册、知识库 CRUD、文档上传/详情/删除/重试、知识库问答、搜索、后台用户/知识库/文档、AI 配置、Prompt、日志、公告等接口大多已有前后端对应。
2. **真正影响联调的是参数和返回结构不一致**：尤其是文档列表、后台文档列表、后台知识库筛选、后台任务重试、系统参数配置、Prompt 模板配置。
3. **前端有些 API 已封装但页面主链路未使用**：例如指定文档问答、多知识库问答、Prompt 详情、系统参数删除等。
4. **PRD 的 MVP 要求里，后台配置、任务管理、日志审计是重点补齐项**，目前接口层有骨架，但字段契约和页面展示还需要统一。

建议优先顺序：

```text
P0：先修会导致接口报错或页面保存失败的问题
P1：再修列表筛选、详情字段、展示字段不一致问题
P2：最后补扩展入口、日志字段别名、增强功能
```

---

## 1. PRD 对本轮联调的要求摘要

PRD 第一版 MVP 的核心目标是完成以下闭环：

```text
注册/登录
  -> 创建知识库
    -> 上传文档
      -> 文档解析与向量化
        -> 基于知识库提问
          -> 查看 AI 回答和引用来源
            -> 管理知识库/文档/问答记录
```

后台 MVP 要求包括：管理员登录、首页数据看板、用户管理、知识库管理、文档管理、AI 模型配置、Prompt 模板管理、文档处理任务管理、系统参数配置、登录日志、操作日志、AI 调用日志。

因此，本轮静态分析重点不是扩展功能，而是检查：

- 前端页面是否调用了接口；
- 后端 Controller 是否实现同路径、同方法接口；
- 请求参数是否能被后端接收；
- 返回字段是否能支撑前端页面展示；
- 哪些问题会阻塞 MVP 演示。

## 6. P0 / P1 / P2 开发优先级

### P0：必须先修，避免页面直接报错或核心功能不可用

#### P0-1：补任务级重试接口

问题：前端调用 `POST /admin/document-tasks/{id}/retry`，后端当前 Controller 未看到该接口。

建议改后端：

```java
@PostMapping("/document-tasks/{id}/retry")
public ApiResponse<DocumentTaskVO> retryTask(@PathVariable Long id) {
    return ApiResponse.ok(adminService.retryTask(id));
}
```

同时 Service 层需要：

```text
1. 根据 taskId 查询任务；
2. 校验任务状态是否 FAILED；
3. 清理旧失败信息；
4. 重新触发文档解析或向量化；
5. 返回最新 DocumentTaskVO。
```

#### P0-2：修 `GET /documents` 参数契约

问题：前端允许不传 `knowledgeBaseId`，并传 `parseStatus/embeddingStatus`；后端当前要求 `knowledgeBaseId` 必填，且不接收状态筛选。

建议改后端：

```java
@GetMapping
public ApiResponse<PageResponse<DocumentVO>> page(
    @RequestParam(required = false) Long knowledgeBaseId,
    @RequestParam(defaultValue = "") String keyword,
    @RequestParam(required = false) DocumentParseStatus parseStatus,
    @RequestParam(required = false) EmbeddingStatus embeddingStatus,
    @RequestParam(defaultValue = "1") int pageNo,
    @RequestParam(defaultValue = "10") int pageSize
) {
    return ApiResponse.ok(documentService.page(knowledgeBaseId, keyword, parseStatus, embeddingStatus, pageNo, pageSize));
}
```

#### P0-3：统一系统参数配置接口

当前前端 `SystemSettingView.vue` 按对象读取和保存：

```ts
{
  maxFileSize,
  allowedTypes,
  chunkSize,
  chunkOverlap,
  topK,
  threshold
}
```

但后端返回的是 `List<ConfigVO>`，保存是 `SystemConfigRequest`。

推荐改后端，提供结构化 DTO：

```java
public record SystemConfigBundleVO(
    Integer maxFileSize,
    List<String> allowedTypes,
    Integer chunkSize,
    Integer chunkOverlap,
    Integer topK,
    BigDecimal threshold
) {}
```

接口保持：

```text
GET  /admin/config/system       -> 返回 SystemConfigBundleVO
POST /admin/config/system       -> 接收 SystemConfigBundleRequest
```

#### P0-4：统一 Prompt 模板字段

前端使用：

```ts
{name, type, content, enabled}
```

建议后端不要复用通用 `ConfigVO`，改为单独的 Prompt VO：

```java
public record PromptTemplateVO(
    Long id,
    String name,
    String type,
    String content,
    Boolean enabled,
    LocalDateTime createTime,
    LocalDateTime updateTime
) {}
```

这样前端不需要大量字段适配，页面也更符合 PRD 的“模板名称、使用场景、启用状态、更新时间”。
